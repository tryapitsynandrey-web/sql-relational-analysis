# Architecture Design Document

## 1. High-Level Architecture Overview
This is a lightweight, SQL-first analytics repository built entirely natively within PostgreSQL 15+. The architecture is designed for simplicity, speed, and reproducibility, enabling an end-to-end data pipeline to execute from raw local flat files (CSVs) to finalized visualization reports entirely via a minimal Docker container and Jupyter notebooks, well within a 10-minute turnaround. It enforces a strict separation of concerns through explicit directory layers: DDL schema definition, fast raw ingestion, automated data quality testing, semantic view virtualization, declarative analytical queries, and a programmatic insights presentation layer.

## 2. Component Responsibility Table
| Area | Responsibility | Dependencies |
| --- | --- | --- |
| **Data/DDL** (`sql/ddl/`) | Target schema definitions, empty table structure instantiation, column data typing, indexes, and constraint additions (PKs/FKs). | None |
| **Load** (`sql/load/`) | Bulk ingestion interface. Uses performant client-side `psql \copy` commands to load raw structural data into instantiated tables. | Data/DDL, `data/raw/olist/*` |
| **Quality Tests** (`sql/tests/`) | Structural validation verifying referential integrity completeness, missing values, empty fields, and base metric rationality checks. | Load |
| **Semantic Views** (`sql/views/`) | The business logic layer. Abstracting normalized raw tables into cohesive fact and dimension virtual tables to prevent duplicated logic in final queries. | Data/DDL |
| **Analysis Queries** (`sql/analysis/`) | Pure SQL declarations answering the fundamental business questions (e.g., retention, logistics SLA delay rates, review correlations). | Semantic Views |
| **Insights Layer** (`analysis/notebooks/`, `analysis/reports/`) | Environment-aware Python execution of the SQL analytics scripts. Maps query results to pandas DataFrames and generates standard MATPLOTLIB visual figures. Output Markdown reports summarize the data into actionable business recommendations. | Analysis Queries |
| **Infra & orchestration** (`Dockerfile`, `.env.example`, `docs/runbook.md`) | Spin-up runtime configurations for the host PostgreSQL engine and the operational instructions for maintaining the pipeline dependencies and sequential execution order. | None |

## 3. Proposed Folder Structure
```text
.
├── .env.example                # Shared connection variable templates (User, Pass, DB, Port)
├── .gitignore                  # Data ignores (must omit data/raw/** to prevent massive git blobs)
├── Dockerfile                  # Minimal PostgreSQL 15+ container runtime
├── README.md                   # Core introduction and Insights layer runbook
├── analysis/                   # Presentation and Execution Layer
│   ├── figures/                # Exported PNG visualizations generated by Python
│   ├── notebooks/              # Jupyter .ipynb connecting to DB & running sql/analysis
│   └── reports/                # Markdown strategic business report write-ups
├── data/                       
│   └── raw/                    # [Ignored] Source CSV files
├── docs/                       # Project Documentation
│   ├── architecture.md         # Architecture overview (This document)
│   ├── dialect_notes.md        # PostgreSQL native syntax decisions
│   ├── recommendations.md      # Consolidated master business strategies
│   └── runbook.md              # Pipeline orchestration step-by-step
└── sql/                        # SQL Execution Pipeline
    ├── analysis/               # Core business questions
    ├── ddl/                    # Create Schema, Tables, Indexes, Constraints
    ├── load/                   # psql \copy statements
    ├── tests/                  # Integrity validations
    └── views/                  # Denormalized fact tables
```
*Note: Configs & Secrets are defined strictly via `.env` (ignored by git) and templated in `.env.example`. Connection strings for Jupyter dynamically resolve `.env` variables or fallback to assumed local PostgreSQL defaults.*

## 4. Data Flow Narrative
1. **Infra Bootstrap**: The OS spins up a background PostgreSQL Docker container strictly from the `Dockerfile`. An authorized user connects via `psql`.
2. **Schema Instantiation**: Execution of `sql/ddl/01` and `02` constructs the structural foundation (schemas and unconstrained tables).
3. **Bulk Ingestion**: `sql/load/01_bulk_insert_data.sql` sweeps through `data/raw/olist/*.csv` and streams thousands of rows straight into the DB memory using `\copy` routines for speed.
4. **Relational Binding**: Execution of `sql/ddl/03_add_constraints.sql` post-ingestion allows the DB to apply primary keys, validate foreign keys, and generate secondary B-tree indexes natively to ensure logical consistency and read performance without grinding the bulk inserts to a halt.
5. **Quality Assurance**: `sql/tests/*.sql` scripts are manually run to verify zero orphaned rows and standard boundaries.
6. **Virtualization**: Execution of `sql/views/*.sql` constructs reusable fact structures (Order Fact, Item Fact) calculating sub-components like delivery SLA dates without persisting redundant data storage.
7. **Execution of Python Insights**: User boots the Python environment. Executing `analysis/notebooks/*.ipynb` dynamically reads the pure SQL from `sql/analysis/`, connects via `psycopg2`, executes it, and renders PNG matplotlib charts to `analysis/figures/`.
8. **Business Value Export**: Stakeholders synthesize chart visuals via static `analysis/reports/*.md`.

## 5. Risks and Trade-offs (Decision Traceability)
### Rejected Alternative: dbt (Data Build Tool) + docker-compose orchestration
- **Reason**: While normally standard for modern analytics, `dbt` and `docker-compose` force complex installation parameters on independent reviewers. The objective is strict <10-minute setup capability; abstracting pure SQL under dbt limits ad-hoc verifiability. 
- **Impact**: Pipeline execution is relegated to a manual runbook order instead of a scheduled automated DAG network.

### Explicit Risks
- **Brittle Ingestion Pipeline**: 
  - *Risk*: `psql \copy` inherently dumps failure traces if upstream CSV schemas mutate (e.g., column addition, data type violations).
  - *Mitigation*: Hardcoded DDL definitions enforce rigidity. Manual `sql/tests/` attempt to catch post-insert logical anomalies.
- **Notebook Execution Drift**:
  - *Risk*: The Jupyter notebooks parse `sql/analysis/*.sql` as raw text arrays. If an engineer updates the SQL to alter a column alias, the pandas dataframe transformation in the notebook immediately throws `KeyError`.
  - *Mitigation*: Strict architectural boundary separation.
- **Line Count Maintenance**:
  - *Rule*: Any single SQL script or Python file exceeding 700 lines strictly fractures readability. All SQL and module files are mandated to break by conceptual grain (e.g. keeping Notebook sizes small by delegating all data wrangling functions to virtualized SQL views).

## SELF-REVIEW STEP
- **The weakest part of the architecture**: Orchestration & Execution Tracking. Relying blindly on a human operator to follow `docs/runbook.md` sequence perfectly guarantees eventual human execution error or missing dependency initialization.
- **Mitigation / Why it is Acceptable**: The explicit requirement is zero dependency, local testable reproducibility within 10 minutes. By eliminating heavy operational components like Airflow, Prefect, or Makefiles, we dramatically reduce the knowledge floor required to validate the pipeline results. Wait-time shrinks toward purely raw Postgres I/O bounds.
